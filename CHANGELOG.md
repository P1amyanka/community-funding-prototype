# Changelog

## Prototype v0.4 / allocation-v0.2 — 2026-07-24

### Раунди та сторінка учасника
- Відновлено кнопку «Новий раунд» після будь-якого завершення.
- Новий раунд успадковує назву, опис, бюджет, кількість учасників і платіжні реквізити.
- Перед відкриттям нового раунду менеджер може встановити новий дедлайн.
- Посилання менеджера та учасників залишаються незмінними між раундами.
- Після відкриття нового раунду те саме посилання учасника знову показує порожню форму голосування.
- Завершені раунди зберігаються в історії та доступні для завантаження у CSV.
- Сторінка учасника перевіряє стан раунду кожні п’ять секунд і оновлює результат без ручного перезавантаження.

### Внесок учасника
- Учасник упізнається в тому самому браузері за локальним приватним ідентифікатором.
- У відкритому раунді внесок можна змінити; у завершеному — ні.
- Новий раунд відкривається з порожньою сумою та потребує нового подання.
- Після успішного завершення учасник бачить свій розрахований внесок.
- Якщо мету не досягнуто, учасник бачить точну суму дефіциту.

### Оплата
- Додано необов’язкове поле «Посилання на Банку, Конверт або номер картки».
- Поле приймає довільний текст без URL-валідації.
- Значення з `http://` або `https://` показується як активне посилання.
- Інші значення, зокрема номер картки, показуються з кнопкою «Копіювати».

### Технічні зміни
- Додано окрему схему prototype v0.4 і RPC для стабільних ініціатив, повторних раундів, історії та редагування внеску.
- Алгоритм розподілу лишається `allocation-v0.2`.

## Prototype v0.3 / allocation-v0.2 — 2026-07-22

### Інтерфейс
- Головну сторінку перейменовано на «Створіть ініціативу».
- Оновлено пояснення процесу для менеджера та учасників.
- Поля «Опис» і «Бюджет» отримали нові приклади.
- Після створення ініціативи одразу відкривається кабінет менеджера.
- Прибрано проміжний екран «Ініціативу створено».
- Сторінку керування перейменовано на «Кабінет менеджера».
- Технічні статуси замінено зрозумілими користувацькими формулюваннями.
- Прибрано пояснення внутрішньої роботи алгоритму й текст про автоматичне закриття.
- Дедлайн не показується, якщо його не встановлено.
- Для учасника залишено одне поле: «Максимальна сума внеску» без додаткової підказки.
- Поточний візуальний стиль збережено.

### Алгоритм allocation-v0.2
- Прибрано мінімальну суму, яку раніше вводив учасник.
- Кожен учасник задає лише максимальну суму внеску.
- Система розраховує найменший необхідний внесок для кожного за принципом рівномірного розподілу з індивідуальними верхніми межами.
- Якщо рівна частка перевищує максимум учасника, його внесок фіксується на максимумі, а залишок порівну перерозподіляється між іншими.
- Розрахунок виконується в копійках, а залишок від округлення розподіляється по одній копійці.
- Якщо сума максимумів менша за ціль, система показує дефіцит.
- CSV оновлено: максимум, розрахований внесок, версія алгоритму.

### Сумісність
- Для сумісності з чинним Supabase RPC у `p_min_amount` передається `0`; нове поле мінімуму в інтерфейсі відсутнє.
